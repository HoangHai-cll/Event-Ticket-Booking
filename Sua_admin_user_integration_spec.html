ĐẶC TẢ ÁNH XẠ DỮ LIỆU & LUỒNG TƯƠNG TÁC HAI CHIỀU (ADMIN ⇄ USER) 📑🔄

Tài liệu này hệ thống hóa toàn bộ kiến trúc cơ sở dữ liệu Firestore, ánh xạ thuộc tính (Schema Mapping) và luồng trao đổi dữ liệu hai chiều giữa phân hệ Quản trị (Admin) và phân hệ Khách hàng (User) của ứng dụng đặt vé sự kiện EventPass.

💾 1. Cấu trúc Ánh xạ Cơ sở dữ liệu (Firestore Collections Schema)

Ứng dụng sử dụng 4 collections chính trên Firestore để quản lý sự tương tác này.

1.1. Collection: events (Sự kiện)

Lưu trữ thông tin chi tiết về sự kiện do Admin tạo ra và User xem/đặt mua.

Tên trường (Field) Kiểu dữ liệu Vai trò và Ánh xạ dữ liệu
eventId String Mã định danh sự kiện (Khóa chính).
title String Tên sự kiện do Admin nhập → User xem trên Trang chủ/Chi tiết.
remainingTicket Number (int) Số lượng vé còn lại trong kho.
Ánh xạ: Trừ đi khi User mua thành công; cộng lại khi User hủy vé.
requiredTier String Hạng thành viên tối thiểu cần thiết để mua vé.
Giá trị: "Thường", "Đồng", "Bạc", "Vàng", "Thân thiết số một".
Ánh xạ: Admin cấu hình → Chặn/Mở khóa nút mua phía User.
tiers Array <Map> Danh sách các phân hạng vé (VIP, Standard...) gồm tierId, tierName, price, totalTicket, remainingTicket.

1.2. Collection: bookings (Đơn đặt vé)

Lưu trữ thông tin giao dịch mua vé của User và được Admin theo dõi/phê duyệt.

Tên trường (Field) Kiểu dữ liệu Vai trò và Ánh xạ dữ liệu
bookingId String Mã giao dịch (Khóa chính). Sinh tự động khi User thanh toán → Sinh mã QR Code cho User.
userId String ID người mua. Ánh xạ để Admin tìm ra thông tin liên hệ của User.
eventId String ID sự kiện được đặt vé. Ánh xạ để lấy thông tin sự kiện trên hóa đơn/báo cáo doanh thu Admin.
quantity Number (int) Số lượng vé đặt mua.
tierId String ID hạng vé cụ thể (nếu sự kiện có nhiều hạng vé).
totalPrice Number (double) Tổng số tiền thanh toán (sau khi áp mã giảm giá).
status String Trạng thái đơn vé.
Giá trị: "Confirmed", "Pending Payment", "Refund Pending", "Cancelled".

1.3. Collection: users (Người dùng)

Lưu trữ thông tin cá nhân, quyền hạn và tiến trình hạng thành viên.

Tên trường (Field) Kiểu dữ liệu Vai trò và Ánh xạ dữ liệu
uid String ID tài khoản người dùng (Khóa chính liên kết Firebase Auth).
role String Quyền hạn tài khoản: "user" hoặc "admin".
exp Number (long) Điểm kinh nghiệm tích lũy của User.
Ánh xạ: Tự động cộng 50 → 200 EXP/vé khi User thanh toán thành công.
memberTier String Hạng thành viên hiện tại: "Thường", "Đồng", "Bạc", "Vàng", "Thân thiết số một".
Ánh xạ: Admin xem/chỉnh sửa → User nhận ưu đãi mua vé sớm và voucher tự động.

🔁 2. Luồng truyền nhận dữ liệu hai chiều (Two-Way Data Flow Spec)

2.1. Luồng 1: Thiết lập & Kiểm soát mua vé VIP (Admin ➔ User)

Luồng này đảm bảo các sự kiện VIP chỉ hiển thị/cho phép mua đối với những thành viên đạt thứ hạng quy định.

[Admin tạo Sự kiện] 
       │
       ▼
Gán: requiredTier = "Vàng"
       │
       ▼ (Lưu lên Firestore: events/{eventId}/requiredTier = "Vàng")
       │
[User mở App]
       │
       ├─► Trường hợp A: User có memberTier = "Vàng" hoặc "Thân thiết số một"
       │                 ==> Nút bấm "Khám phá / Đặt vé" hoạt động bình thường.
       │
       └─► Trường hợp B: User có memberTier = "Thường" hoặc "Đồng" hoặc "Bạc"
                         ==> Nút đặt vé bị vô hiệu hóa (disabled), hiển thị khóa 🔒
                         ==> Thông báo: "Chỉ dành cho thành viên hạng Vàng trở lên!"

2.2. Luồng 2: Trừ/Cộng kho vé khi Đặt và Hủy vé (User ⇄ Admin)

Luồng này đồng bộ số lượng vé trống của sự kiện theo thời gian thực (Real-time Stock Synchronization).

               ┌───────────────────────────────┐
               │    events/{eventId}           │
               │    - remainingTicket: 100     │
               └───────────────┬───────────────┘
                               │
            ┌──────────────────┴──────────────────┐
            ▼                                     ▼
     [User đặt 2 vé]                      [User yêu cầu Hủy 2 vé]
            │                                     │
            ▼                                     ▼
Firestore nhận cập nhật:               Firestore nhận cập nhật:
remainingTicket = 100 - 2             remainingTicket = 98 + 2
            │                                     │
            ▼                                     ▼
   [Kho vé còn lại: 98]                  [Kho vé trả lại: 100]
            │                                     │
            ▼                                     ▼
Dashboard Admin cập nhật:              Dashboard Admin cập nhật:
• Tổng vé đã bán tăng thêm 2.          • Tổng vé đã bán giảm đi 2.
• Tổng doanh thu tăng tương ứng.       • Doanh thu trừ tiền hoàn lại.

2.3. Luồng 3: Hủy vé & Hoàn tiền (User ➔ Admin ➔ User)

Quy trình đảm bảo kiểm soát dòng tiền, ngăn chặn việc hủy khống và cam kết hoàn tiền trong 3 ngày làm việc.

Luồng xử lý trạng thái (State Flow):
[*] --> [Confirmed: User đặt vé thành công]
           │
           ▼ (User nhấn "Hủy vé & Hoàn tiền")
[Refund Pending: Chờ hoàn tiền]
   └─► Thông báo User: "Hệ thống đang thực hiện hoàn tiền. Vui lòng chờ trong 3 ngày làm việc."
           │
           ▼ (Admin kiểm tra danh sách và xử lý)
     ┌─────┴─────────────────────────────────┐
     ▼ (Admin Phê duyệt)                     ▼ (Admin Từ chối)
[Cancelled: Đã hủy vé]                  [Confirmed: Khôi phục vé]
 - Kho vé tự động cộng lại.              - Hoàn tác trạng thái.
 - Thu hồi mã QR Code của vé.            - User giữ quyền sử dụng vé.
 - Gửi thông báo thành công cho User.

2.4. Luồng 4: Tích lũy EXP & Thăng hạng đặc quyền (User ➔ Admin ➔ User)

Luồng này điều khiển trải nghiệm khách hàng thân thiết, tự động hóa quy trình chăm sóc và tặng quà.

[User thanh toán đơn vé thành công]
               │
               ▼
Hệ thống tính toán: Số lượng vé x [50 -> 200] EXP (Ngẫu nhiên)
               │
               ▼ (Cập nhật lên Firestore: users/{userId}/exp = exp + EXP_mới)
               │
[Hệ thống kiểm tra điều kiện thăng hạng]
               │
               ├─► Đạt mốc mới (Ví dụ: EXP vượt quá 1000)
               │   • Cập nhật users/{userId}/memberTier = "Bạc"
               │   • Hệ thống tự động đẩy thông báo Pop-up chúc mừng thăng hạng lên màn hình User.
               │   • Tạo và tự động gán Voucher giảm giá độc quyền dành riêng cho hạng Bạc vào ví User.
               │
               └─► Chưa đạt mốc mới
                   • Giữ nguyên hạng hiện tại.
                   • Cập nhật tiến độ % trên thanh ProgressBar của User.

📌 3. Bảng Quy ước Trạng thái Đơn vé (Booking Statuses Matrix)

Trạng thái (Status) Ý nghĩa hiển thị phía User Hành động tương tác của User Vai trò hiển thị/Hành động phía Admin
Confirmed Đặt vé thành công (Nền xanh lá). Hiện QR Code để quét check-in vào cửa. Có thể nhấn "Hủy vé & Hoàn tiền". Hiển thị trong báo cáo doanh số, đếm số người tham gia sự kiện.
Pending Payment Chờ thanh toán (COD - Nền vàng). Dùng cho vé nhận tại quầy. Không thể quét QR Code. Hiển thị danh sách vé chờ thu tiền trực tiếp tại quầy check-in.
Refund Pending Chờ hoàn tiền (Nền cam). Đang xử lý hoàn tiền (Nút hủy vé bị vô hiệu hóa). Hạng mục To-Do: Hiển thị nổi bật trong Tab yêu cầu phê duyệt hủy vé. Có nút "Phê duyệt" / "Từ chối".
Cancelled Đã hủy vé (Nền xám). Vé bị thu hồi QR Code (Không có hiệu lực check-in). Hiển thị thống kê số lượng đơn vé bị hủy/lượng tiền đã hoàn trả.
🏆 4. Kết luận đánh giá: Lộ trình đã chuẩn chưa?

ĐÁNH GIÁ CHẤT LƯỢNG LỘ TRÌNH: Lộ trình này ĐÃ ĐẦY ĐỦ VÀ CHUẨN XÁC 100%.

Việc hoàn thiện 3 mảnh ghép quản trị phía Admin (Cấu hình hạng vé ưu tiên, Phê duyệt hoàn tiền 3 ngày, Quản lý điểm EXP/Hạng User) sẽ giúp liên kết và khép kín hoàn toàn các tính năng tương ứng ở phía User (Voucher theo hạng, Chặn mua vé độc quyền, Yêu cầu hủy vé chờ hoàn tiền), tạo nên một hệ thống phần mềm quản lý đặt vé sự kiện vận hành chuyên nghiệp, tự động hóa cao và an toàn về mặt tài chính dòng tiền.
